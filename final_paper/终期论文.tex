\vspace{0.5cm}

本文介绍了一个面向电视剧《甄嬛传》的 AI 角色语音对话系统------\,\textbf{甄嬛传\,\textperiodcentered\,千声千面}。该系统从 76 集原剧音频中自主构建了包含 14 个角色、13,605 段语音片段（总时长约 16.7 小时）的高质量中文语音数据集，并基于 CosyVoice3-0.5B（Zero-shot 语音克隆）和 GPT-SoVITS（微调语音合成）双引擎实现了多角色语音克隆与情感语音合成。系统结合大语言模型（LLM）实现角色性格建模与对话生成，支持单人智能对话与双人即兴对戏两种交互模式，并通过 SSE 流式传输与 Web Audio API 实现了低延迟的语音播放体验。实验表明，CosyVoice3 Zero-shot 方案在音色还原度和自然度上显著优于微调方案，创新性的\textbf{情绪 Speaker}方案有效解决了情绪控制与音色保真之间的矛盾。

\section{选题背景}
\label{sec:background}

\subsection{研究背景}

语音合成（Text-to-Speech, TTS）与语音克隆（Voice Cloning）技术在近年来取得了飞速发展。以 VALL-E、CosyVoice、GPT-SoVITS 等为代表的新一代语音合成模型，凭借强大的零样本（Zero-shot）学习能力和细粒度的韵律控制，使得高质量的个性化语音合成成为可能。与此同时，大语言模型（LLM）在自然语言理解与生成方面的突破，为构建具有角色个性的智能对话系统提供了新思路。

影视IP的语音交互应用是上述技术的典型落地场景。《甄嬛传》作为中国最具影响力的古装电视剧之一，拥有大量经典角色和丰富的台词素材，其角色声音具有高辨识度和情感表现力，是语音克隆技术的理想实验对象。然而，将语音克隆技术应用于多角色场景面临若干挑战：首先，从原始音频中提取并标注多角色数据需要完整的音频处理流水线；其次，多角色联合训练容易因数据不均衡导致过拟合；最后，在保持音色保真的同时实现情绪控制是一个尚未充分解决的问题。

\subsection{课程要求}

本项目为\textbf{语音信息处理}课程大作业，要求涵盖语音合成、语音克隆、说话人识别、情感识别等核心技术，具备完整的算法实现、自主构建数据集的能力，并展示创新应用场景。

\section{实验目标}
\label{sec:objectives}

本项目旨在构建一个面向《甄嬛传》的 AI 角色语音对话系统，实现以下核心目标：

\begin{enumerate}[leftmargin=*]
    \item \textbf{自主构建多角色语音数据集}：从 76 集原剧音频出发，通过人声分离、语音识别、声纹聚类与人工标注，构建包含 14 个角色的高质量中文语音数据集。
    \item \textbf{多角色语音克隆与合成}：基于 CosyVoice3-0.5B（Zero-shot）和 GPT-SoVITS（微调）双引擎实现角色声音的高保真复现，支持用户实时对比两种引擎的合成效果。
    \item \textbf{情感语音合成}：实现喜悦、愤怒、悲伤、平静四种情绪的语音合成，在不损失音色保真度的前提下自然表达角色情感。
    \item \textbf{角色智能对话}：结合 LLM 实现角色性格建模，支持\textbf{现代来客}和\textbf{宫廷中人}两种身份的单人对话，以及双角色自动即兴对戏。
    \item \textbf{完整系统集成}：构建包含 React 前端、FastAPI 后端中间层、云端 TTS/ASR/LLM 服务的完整系统，实现流式语音播放和实时交互。
\end{enumerate}

\subsubsection{数据来源与规模}

本项目的数据源为电视剧《甄嬛传》全集 76 集 MP3 音频（128 kbps、44.1 kHz、立体声），已完成去片头片尾处理（无人播剧版），原始总时长约 80 小时。所有数据处理与模型训练均在 AutoDL 云 GPU 实例（RTX 4080 SUPER 单卡，16 GB 显存）上完成。

{\color{Amid}\rule{4pt}{0pt}}\,\textbf{\color{Amid}阶段一：原始数据采集}\\[3pt]
数据来自电视剧《甄嬛传》全部 76 集原声 MP3 文件。音频格式为 44.1 kHz 立体声、128 kbps 比特率，已预先去除每集的片头片尾部分（即无人播剧版本），总计时长约 80 小时。

{\color{Amid}\rule{4pt}{0pt}}\,\textbf{\color{Amid}阶段二：UVR5 人声分离}\\[3pt]
使用 UVR5\cite{uvr5} 的 demucs\cite{defossez2021demucs} 引擎配合 mdx\_extra\_q 预训练权重，对 76 集音频逐一进行人声与背景音乐的分离。该模型在音乐分离基准上对人声轨的分离效果表现优异，尤其对中文语音的中频段保留较为完整。分离后保留纯净人声轨，仍为 44.1 kHz 立体声格式，同时去除了背景音乐、环境音效等非人声成分。

{\color{Amid}\rule{4pt}{0pt}}\,\textbf{\color{Amid}阶段三：VAD 切分 + ASR 转写}\\[3pt]
采用 FunASR\cite{funasr} 的 SeacoParaformer 模型（VAD 与 ASR 二合一）对纯净人声进行语音活动检测（Voice Activity Detection）和自动语音识别（Automatic Speech Recognition）。模型首先检测音频中的语音活动区域，然后将字级别的时间戳合并为句子级片段：当相邻两句话的间隔超过 800 ms 时进行切分，确保每段 WAV 对应一句完整的台词。输出规格为 16 kHz 单声道 WAV，每段时长控制在 2$\sim$ 15 秒之间。本阶段共产生 15,685 段语音片段。

{\color{Amid}\rule{4pt}{0pt}}\,\textbf{\color{Amid}阶段四：声纹提取 + 说话人聚类}\\[3pt]
使用 3D-Speaker\cite{3d-speaker} 中的 eres2netv2 模型对全部 15,685 段 WAV 逐一提取 512 维说话人声纹特征向量（Speaker Embedding），随后使用 scikit-learn\cite{sklearn} 的 KMeans 算法（设置 $n_\text{clusters}=30$）将相似声纹特征的片段自动归入同一簇。聚类结果为 30 个匿名说话人簇（编号 spk\_00 $\sim$  spk\_29）。聚类后片段数最多的 15 个簇的分布如表~\ref{tab:cluster-top15}所示。

{\color{Amid}\rule{4pt}{0pt}}\,\textbf{\color{Amid}阶段五：人工标注 + 数据清洗}\\[3pt]
聚类完成后，从每个说话人簇中随机抽取 5 条音频，由人工逐一试听并根据台词内容、语气特征与剧中角色的说话风格进行综合判断，将匿名簇映射到具体角色。标注过程共经历三轮迭代交叉验证：第一轮成功映射 22/30 簇，第二轮推进至 27/30，第三轮交叉验证并纠正误标后最终确认 24/30 簇。另有 3 个簇（阿晋、净白师太、齐妃）因 KMeans 聚类不准确，簇内混入了多个不同说话人的片段，音色杂乱不可靠，经人工确认后淘汰；剩余簇为空或纯噪声。最终保留 14 个高可信度角色。

\subsubsection{14 角色 Zero-shot Speaker 注册}

从清洗后的 13,605 段数据中，为每个角色精心挑选 1 条高质量的参考音频（5$\sim$ 10 秒、清晰无杂音、最能代表该角色的典型音色），同时匹配对应的台词文本（Prompt Text），通过 CosyVoice3 的 \texttt{add\_zero\_shot\_spk} 接口逐一注册。每个角色的声纹特征（Speaker Embedding）由预训练的 campplus 网络自动提取并持久化保存到 \texttt{spk2info.pt} 文件中，共注册 14 个基础角色 Speaker，覆盖剧中全部可对话角色。

\subsubsection{情绪 Speaker 创新方案}

\paragraph{问题分析}
Zero-shot 方案能够完美复现角色的中性音色，但无法自然地表达喜悦、愤怒、悲伤等情绪变化。CosyVoice3 虽然提供了 Instruct 控制接口，可以通过文字指令驱动模型使用特定情绪说话，但实验发现：Instruct 方式在施加情绪控制的同时会导致音色出现明显损失——情绪控制和音色保真构成一对矛盾，难以兼顾。

\paragraph{解决方案}
提出\textbf{情绪 Speaker}方案：不依赖 Instruct 指令控制情绪，而是从 76 集原剧音频中\textbf{手工筛选}角色在真实情绪状态下的台词片段，将这些情绪音频本身注册为独立的 Zero-shot Speaker。推理时根据前端传入的情绪标签，路由到对应的情绪 Speaker——本质是用该角色真实情绪状态下的音频做声纹提取，走纯 Zero-shot 路径，因此音色完全不丢失。

\paragraph{情绪数据采集}
从 76 集原声中手工截取了 4 个主要角色（甄嬛、华妃、皇上、皇后）$\times$ 4 种情绪（喜悦、愤怒、悲伤、平静）的台词片段，各角色各情绪的采集数量如表~\ref{tab:emotion-data}所示。每条音频通过 FunASR 实际转写为对应台词文本，确保 Prompt Text 与音频内容严格一致（v2 版本改进）。最终共计 107 段情绪音频。

% ── 3.3 GPT-SoVITS 微调 ──────────────────────────────────────────────────
\subsection{GPT-SoVITS 微调}
\label{sec:gptsovits}

GPT-SoVITS\cite{gptsovits} 在本项目中承担备用微调 TTS 引擎的角色。与 CosyVoice3 Zero-shot 方案不同，GPT-SoVITS 需要为每个角色训练独立的 GPT 语义模型和 SoVITS 声学模型权重，再在推理时通过“角色权重 + 参考音频 + 参考文本 + 待合成文本”共同决定最终语音。该模块的目标并非替代 CosyVoice3，而是提供一条可复现的角色级微调路线，并用于和 Zero-shot 方案进行效果对照。

从工程实现看，GPT-SoVITS 微调模块可以被拆分为“数据整理、特征预处理、双模型训练、权重筛选、服务端接入”五个连续步骤。数据整理阶段保证每条音频和台词一一对应；特征预处理阶段把文本和音频转为 GPT-SoVITS 可训练的中间表示；训练阶段分别得到文本到语义 token 的 GPT 权重和语义 token 到声学表现的 SoVITS 权重；权重筛选阶段只保留稳定的最终 epoch；服务端接入阶段再根据角色 ID 与情绪标签加载对应权重和参考音频。这样的拆分使微调工作不只是一次模型训练，而是形成了从数据到网页推理的完整闭环。

\subsubsection{微调数据准备}

GPT-SoVITS 的训练数据直接复用第~\ref{sec:data}节得到的 14 角色清洗数据集。为适配 GPT-SoVITS 的训练入口，系统将每条语音整理为标准 list 格式：

\begin{center}
\fcolorbox{Bmid}{Blight}{%
    \parbox{0.72\textwidth}{\centering\small\ttfamily
    音频绝对路径|说话人|zh|台词文本}}
\end{center}

其中音频路径保留为绝对路径，避免训练脚本在不同工作目录下运行时找不到 WAV 文件；说话人字段使用角色英文 ID；语言字段固定为 \texttt{zh}；文本字段来自数据处理阶段的 ASR 转写与人工清洗结果。随后按角色拆分为 \texttt{\{role\}\_all.list}、\texttt{\{role\}\_train.list} 和 \texttt{\{role\}\_dev.list} 三类文件，分别用于全量参考、训练和验证。该组织方式使每个角色的数据边界清晰，后续训练、推理和问题排查均可直接定位到角色级 list。

项目中实际覆盖的角色包括甄嬛、华妃、皇后、皇上、沈眉庄、安陵容、苏培盛、叶澜依、崔槿汐、温实初、浣碧、果郡王、曹贵人、太后共 14 人。为了兼容网站前端的角色 ID 与 GPT-SoVITS 训练脚本中的英文 ID，服务端还维护了少量别名映射，例如 \texttt{anlinrong} 对应训练侧的 \texttt{anlingrong}，\texttt{yixiu} 对应 \texttt{huanghou}，\texttt{wensichu} 对应 \texttt{wenshichu}。这样前端无需感知训练文件命名差异，后端即可完成角色到模型权重的准确路由。

数据准备阶段还有一个关键设计：list 中的台词文本不仅供训练使用，也在推理时用于反查参考音频对应的 \texttt{ref\_text}。GPT-SoVITS 对参考音频和参考文本的一致性较敏感，如果参考音频真实内容与 \texttt{ref\_text} 不一致，容易造成发音含混、停顿异常或音色漂移。因此，项目在构建 list 时保留了“音频文件名 \(\to\) 台词文本”的可追溯关系，为后续情绪参考音频路由提供了基础。

在具体划分上，\texttt{all.list} 保留该角色的全部可用样本，主要用于参考音频抽样和整体数据检查；\texttt{train.list} 作为模型参数更新的数据来源；\texttt{dev.list} 用于训练过程中的验证和快速试听。由于本项目中的角色数据量并不完全均衡，一些配角的可用语音片段明显少于主角，因此数据准备时更强调“角色边界清晰”和“样本可追溯”，避免为了凑数量把其他角色声音混入训练集。这样虽然牺牲了一部分数据规模，但能减少角色音色污染，更符合角色级语音合成的目标。

\subsubsection{双版本模型权重微调}

本项目没有采用“全部角色混合训练一个模型”的方式，而是对 14 个角色分别训练独立权重。这样可以降低不同角色之间的音色混淆，也便于网站侧根据 \texttt{character\_id} 精确选择模型。每个角色同时训练 \texttt{v4} 与 \texttt{v2ProPlus} 两个版本，共形成 \(14 \times 2 = 28\) 组角色权重。两套版本的定位如表~\ref{tab:gptsovits-versions}所示。

训练脚本 \path{batch_finetune_roles.py} 将 GPT-SoVITS WebUI 中原本需要手工完成的步骤封装为可批量执行的命令行流程：自动发现角色 list，按角色和版本生成实验名，分阶段完成特征提取与训练，并支持跳过已存在的预处理结果。最终使用 \path{run_all_roles_epoch10_final_only.sh} 统一执行收敛版本：SoVITS 与 GPT 均训练 10 个 epoch，并只保留 epoch 10 的最终导出权重。该策略减少了中间 checkpoint 对磁盘的占用，也避免网页接入时出现权重版本混乱。

\texttt{v4} 和 \texttt{v2ProPlus} 的差异不仅体现在权重名称上，也体现在训练方式上。\texttt{v4} 版本使用 \texttt{gsv-v4-pretrained} 系列预训练权重，并通过 \texttt{s2\_train\_v3\_lora.py} 对 SoVITS 部分做 LoRA 微调，项目中设置 \texttt{lora\_rank=32}，希望在角色适配和过拟合控制之间取得平衡。\texttt{v2ProPlus} 版本使用 GPT-SoVITS 的 \texttt{v2ProPlus} 预训练权重与对应配置文件，训练链路更接近常规 SoVITS 微调，可作为部分角色在 \texttt{v4} 效果不稳定时的备选。最终系统保留两套版本，是为了在展示阶段能够按角色比较、按效果回退，而不是被单一模型版本绑定。

每组训练实验都以“角色 ID + 模型版本”作为最小管理单元，例如同一角色会分别生成 \texttt{zhenhuan\_v4} 与 \texttt{zhenhuan\_v2ProPlus} 两条实验记录。训练产物也按版本分目录保存，GPT 权重进入 \texttt{GPT\_weights\_v4} 或 \texttt{GPT\_weights\_v2ProPlus}，SoVITS 权重进入 \texttt{SoVITS\_weights\_v4} 或 \texttt{SoVITS\_weights\_v2ProPlus}。这种目录结构让后端可以直接根据 \texttt{character\_id} 和 \texttt{version} 拼出候选权重路径，也方便后续对单个角色重新训练或替换，而不会影响其他角色。

随后进入两条训练支路。SoVITS 部分更偏向声学建模，负责把语义 token 转换为具有角色音色和韵律的声音，因此它直接影响“像不像这个角色”和声音稳定性；GPT 部分更偏向语义建模，负责根据待合成文本预测合适的语义 token，因此它影响合成语音的断句、节奏和文本跟读能力。只有两部分权重同时匹配同一角色、同一版本，推理时才能得到完整结果。如果只替换其中一侧权重，可能出现音色不一致、发音不清或文本跟随能力下降的问题。

批量训练脚本还加入了可复用和可恢复机制。每次运行时，脚本会先检查 1A、1B、1C 的中间产物是否已经存在，若存在则默认跳过对应预处理，避免重复消耗 GPU 时间；如果数据发生修改，也可以强制重新预处理。训练时脚本按角色和版本逐项执行，并根据设定的 epoch 保存权重。最终使用的训练配置统一为 SoVITS 10 个 epoch、GPT 10 个 epoch，并将保存间隔设为 10，使导出目录中只保留最终 epoch 的 \texttt{.pth} 和 \texttt{.ckpt}。这一策略保证了实验结果简洁稳定，也降低了服务端接入时误选中间权重的风险。

\subsubsection{情绪控制与推理实现}

GPT-SoVITS 本身不稳定支持直接输入 \texttt{emotion=喜悦/愤怒/悲伤/平静} 这类显式情绪参数，因此本项目采用“情绪参考音频”进行间接控制。推理服务收到 LLM 输出的情绪标签后，优先在 \texttt{emotion\_samples/\{character\}/\{emotion\}/} 中随机选择一条真实剧集片段，并根据文件名从角色 list 中反查对应台词作为 \texttt{ref\_text}；若该角色或该情绪没有可用样本，则回退到角色 \texttt{all.list} 中的普通参考音频。参考音频默认限制在 3$\sim$10 秒之间，以避免过短样本缺少韵律信息、过长样本引入多余语义。

推理输入由四部分组成：角色微调权重、参考音频、参考音频真实文本和待合成文本。参考音频负责提供音色与情绪基调，参考文本帮助模型对齐参考音频内容，待合成文本则来自 LLM 生成后的 \texttt{tts\_texts.gpt\_sovits} 字段。为了避免 GPT-SoVITS 将 CosyVoice 控制 token 当作英文读出，服务端会先删除 \texttt{[breath]}、\texttt{[laughter]}、HTML 标签等非中文控制符，并自动补齐中文句末标点。与此同时，LLM prompt 会要求 GPT-SoVITS 文本主要通过逗号、顿号、省略号、问号和叹号控制停顿与语气，使韵律控制保持在模型可稳定理解的范围内。

服务化部署方面，GPT-SoVITS 被封装为独立 FastAPI 服务，对外提供 \texttt{POST /synthesize}。服务默认使用 \texttt{v4} 权重，当推理失败或指定版本不可用时自动回退到 \texttt{v2ProPlus}；同时维护 \((role, version)\) 级别的 TTS 对象缓存，减少连续请求同一角色时反复加载权重的延迟。推理完成后，服务除返回音频路径外，还保存包含角色、版本、参考音频、参考文本和采样参数的 metadata，便于后续复现实验与定位问题。

在一次实际合成请求中，前端或中间层只需要传入角色 ID、待合成文本、情绪标签和可选版本号。GPT-SoVITS 服务内部完成四步处理：首先清洗待合成文本，删除 CosyVoice token 和 HTML 标签；然后根据 \texttt{character\_id} 与 \texttt{emotion} 选择 3$\sim$10 秒的参考音频；接着解析并加载对应角色的 GPT 权重和 SoVITS 权重；最后调用推理脚本生成 WAV 文件并返回路径。该封装把复杂的训练目录、权重路径、参考文本和采样参数隐藏在服务内部，使网站侧只面对统一的 TTS 接口。

这一推理封装也反向验证了前述微调流程的必要性。训练阶段输出的 GPT 权重负责将 LLM 生成的文本转为语义 token，SoVITS 权重负责把语义 token 渲染成角色声音，数据准备阶段保存的 list 又负责提供参考音频与 \texttt{ref\_text}。因此，微调流程中的每一类中间文件都在最终网站调用中有明确用途，而不是只服务于离线训练。通过这种方式，GPT-SoVITS 模块从“能训练出声音”进一步变成了“能被网页稳定调度的角色语音服务”。

% ── 3.4 ASR + LLM 对话模块 ───────────────────────────────────────────────
\subsection{ASR + LLM 对话模块}
\label{sec:asr-llm}

ASR + LLM 模块负责把用户的语音或文字输入转化为角色回复、情绪标签和 TTS 可用文本，是“角色可对话”的核心中间层。该模块以 FastAPI 服务形式独立部署，主要提供 \texttt{POST /asr}、\texttt{POST /generate}、\texttt{POST /generate/duet} 和 \texttt{POST /summarize} 四类接口，供后端网关统一调用。其中 \texttt{/asr} 面向浏览器录音，\texttt{/generate} 面向单人角色对话，\texttt{/generate/duet} 面向双人即兴对话，\texttt{/summarize} 用于结束对话后的态度与总结语生成。这样的拆分使语音识别、角色生成和总结功能彼此独立，又能在中间层组合成完整交互链路。

在实际交互中，该模块承担了“非结构化输入到结构化输出”的转换任务。用户输入可能是一段浏览器录音，也可能是一句手动输入的文字；LLM 原始输出也可能出现格式不稳定、情绪字段缺失或 TTS 文本不合规等情况。因此服务端不会直接把模型输出交给前端和 TTS，而是先通过 prompt 约束生成结构，再用后处理函数对 JSON、情绪和文本进行校验。最终输出被统一整理为展示文本、情绪标签、CosyVoice 文本和 GPT-SoVITS 文本四类信息，供前端展示和语音合成链路复用。

\subsubsection{ASR 模块设计}

ASR 采用 \texttt{faster-whisper} 推理框架，并在部署时配置 \texttt{faster-whisper-small} 或兼容的本地 Whisper 模型路径。选择 small 级别模型的原因在于：本系统面向网页短句对话，单次录音通常只有几秒到十几秒，系统更关注交互延迟与中文短句识别稳定性，而非长音频批处理的极限准确率。与 tiny/base 相比，small 在中文识别上更稳；与 medium/large 相比，它对显存和响应时间更友好。

服务端在首次收到语音识别请求时加载模型，并缓存为全局对象，后续请求直接复用，避免每次录音都重复初始化模型。识别时固定设置 \texttt{language=zh}，防止短句输入被误判为其他语言；同时开启 \texttt{vad\_filter=True} 过滤静音和空白片段，降低浏览器录音前后空白对识别结果的影响。接口返回识别文本、语言和语言置信度。前端交互上，ASR 结果不会自动发送给 LLM，而是先回填到输入框，由用户确认或修改后再发送，这一设计将语音识别错误的影响控制在对话生成之前。

从接口实现看，\texttt{/asr} 接收 multipart 表单中的 \texttt{audio} 文件，先写入临时文件，再交给 faster-whisper 识别，最后在 \texttt{finally} 块中删除临时文件，避免服务端堆积录音缓存。如果未配置 \texttt{ASR\_MODEL} 或 \texttt{ASR\_MODEL\_PATH}，接口会返回 501 错误，但文字输入和 LLM 对话功能仍可正常使用，保证系统在 ASR 环境缺失时仍能演示核心角色对话能力。

ASR 模块在系统中并不直接决定角色回复，而是提供一个可被用户确认的输入增强能力。这样设计的原因是语音识别错误往往会被 LLM 放大：如果错误文本直接进入角色 prompt，后续回复、情绪判断和 TTS 都会基于错误语义继续生成。通过“识别结果回填输入框、用户确认后再生成”的交互方式，系统把 ASR 风险控制在第一步，同时保留了语音输入的便利性。

\subsubsection{LLM 对话 Prompt 构建}

LLM 单人对话采用 system prompt 与 user prompt 分层设计。system prompt 负责定义模型身份和输出约束，要求模型作为《甄嬛传》角色对话生成器，只输出 JSON，并包含 \texttt{text}、\texttt{emotion} 和 \texttt{tts\_texts} 三个顶层字段。其中 \texttt{text} 用于前端气泡展示，\texttt{emotion} 必须从喜悦、愤怒、悲伤、平静四类中选择，\texttt{tts\_texts} 则同时包含面向 CosyVoice 和 GPT-SoVITS 的两套文本。user prompt 注入当前对话角色、角色风格、用户身份、最近对话历史、用户本轮输入和用户指定情绪，使生成结果同时受角色人设、上下文和交互模式约束。

单人对话的输出字段被设计为“一次生成，多处复用”：\texttt{text} 保持自然、干净，直接进入前端聊天气泡；\texttt{emotion} 同时驱动前端视觉反馈和后端情绪音频路由；CosyVoice 专用文本与 GPT-SoVITS 专用文本分别交给两种 TTS 引擎。这样可以避免“先生成回复，再二次调用 LLM 生成 TTS 文本”的额外延迟，也减少多轮生成导致的语义不一致。

为了让模型更稳定地输出可解析 JSON，prompt 中还对禁止内容做了明确限制：不能输出 Markdown 代码块，不能添加解释性文字，不能使用旁白、舞台动作、表情符号或角色名前缀。对于宫廷语气的表达，prompt 不依赖固定模板，而是通过角色风格描述、用户身份和最近历史共同约束，使回复既能保持人物口吻，又不会完全变成重复台词。服务端随后会再次剥离可能残留的代码块标记，并在 JSON 解析失败时进入 fallback 逻辑，保证前端不会因为单次 LLM 输出异常而中断。

项目中单人对话有两种身份入口。第一种是\textbf{现代来客}，即用户以外部访客身份与角色交流。此时 user prompt 中写入“用户身份：现代来客”，角色回复更偏向解释、交流和剧情问答，适合用户询问宫廷生活、角色经历或人物感受。第二种是\textbf{宫廷中人}，即用户选择扮演剧中某一角色再与目标角色对话。此时 prompt 写入“用户身份：宫廷中人：某角色”，LLM 需要根据双方身份调整称呼、权力关系、亲疏远近和情绪张力。例如，皇上与甄嬛对话时会出现帝王口吻和君臣关系，华妃与甄嬛对话时则更容易生成试探、讥讽或对峙语气。

\subsubsection{双人即兴对话}

双人即兴对话不是一次性生成整段剧本，而是逐轮生成当前发言角色的一句台词。每轮调用时，user prompt 都会传入当前发言角色、当前角色风格、对方角色、对方角色风格、场景或话题，以及已经生成的完整对话历史。system prompt 进一步约束输出只能包含 \texttt{text} 与 \texttt{emotion}，且 \texttt{text} 只能是台词本身，不能带角色名、引号、旁白、动作描写或舞台说明。后端在每轮生成后将结果追加到 history，并交换当前角色与对方角色，直到达到设定轮数或用户主动停止。

这种“逐轮生成 + 完整历史传递”的方式有两个优点：一是每句台词都能接住上一句，不会像整段生成那样容易后半段失控；二是两个角色的人设分别注入，发言风格更容易保持差异。前端可在每轮生成后立即展示文字并触发 TTS 合成，用户看到的是连续自动对戏，而不是等待完整剧本一次性返回。

双人即兴对话还对台词长度与格式做了更强限制：文本长度控制在 20 到 50 个汉字左右，只允许输出当前角色的一句或两句自然台词，不允许输出角色名、引号、括号中的动作或旁白。后处理阶段会进一步清除可能误生成的“角色名：”前缀，避免前端展示和 TTS 朗读时把角色名也读出来。相比单人对话，这一模式更强调节奏和角色间的接话关系，因此每一轮都携带完整历史，而不是只传最近一句。

\subsubsection{LLM 输出校验与双 TTS 文本策略}

LLM 输出后会经过服务端校验和归一化。首先，服务会剥离可能出现的 Markdown 代码块，解析 JSON；若 LLM 调用失败或返回结构异常，则使用本地 fallback 话术保证网站可演示。随后，\texttt{emotion} 会被归一化到四类合法情绪中；如果用户手动指定情绪，则以用户指定值为准。展示文本会删除副语言 token、HTML 标签和多余空白，防止前端气泡出现控制符或不自然文本。

校验逻辑主要分为三层。第一层是结构校验，检查返回值是否能解析为 JSON，是否包含 \texttt{text}、\texttt{emotion} 和 \texttt{tts\_texts}；第二层是语义校验，确保情绪字段落在喜悦、愤怒、悲伤、平静四类中，并在用户指定情绪时优先服从用户输入；第三层是语音文本校验，分别检查 CosyVoice 和 GPT-SoVITS 的文本是否与展示文本语义一致、是否包含不被对应引擎支持的控制符。三层校验使 LLM 的开放式生成结果能够转化为后端可执行的稳定接口数据。

双 TTS 文本策略是本模块的一个关键设计。CosyVoice 支持少量副语言 token，因此 \texttt{tts\_texts.cosyvoice} 可以在不改变语义的前提下保留 \texttt{[breath]}、\texttt{[laughter]}、\texttt{[sigh]}、\texttt{[cough]}、\texttt{[lipsmack]}、\texttt{[noise]} 以及 \texttt{<strong>} 强调标签；服务端会限制 token 必须来自白名单、数量最多两个，并检查去除 token 后的文本是否仍与展示文本一致。GPT-SoVITS 不支持这些控制符，因此 \texttt{tts\_texts.gpt\_sovits} 会被清洗为干净中文文本，主要依赖中文标点表达停顿和语气。这样同一次 LLM 生成即可同时服务两个 TTS 引擎，既保留 CosyVoice 的细粒度控制，又避免 GPT-SoVITS 误读控制符。

例如前端展示文本可以是“本宫今日心情尚可，你倒是有眼力见儿。”；若情绪为喜悦，CosyVoice 文本可扩展为“\texttt{[laughter]}本宫今日心情尚可，你倒是有眼力见儿。”，利用笑声 token 增强语气；而 GPT-SoVITS 文本则保持为“本宫今日心情尚可，你倒是有眼力见儿。”，只通过逗号和句号控制停顿。若情绪为悲伤，CosyVoice 可在句首加入 \texttt{[sigh]}，GPT-SoVITS 则可用省略号和短句拆分表达犹豫与停顿。两套文本语义必须一致，差别只体现在控制方式上。

\section{数据集构建结果}
\label{sec:results}
\label{sec:result-data}

经过第~\ref{sec:data}节所述的完整数据处理流水线，成功从《甄嬛传》76 集原剧音频中自主构建了包含\textbf{14 个角色、13,605 段语音片段、总时长约 16.7 小时}的高质量中文语音数据集。最终数据分布如表~\ref{tab:data-dist} 及图~\ref{fig:pie-dist} 所示。

数据清洗效果显著（表~\ref{tab:data-clean}）：四轮过滤共剔除 2,080 段低质量数据（占比 13.3\%），其中角色过滤阶段丢弃量最大（$-$1,682 段），主要来自非目标角色片段和背景杂音；文本过滤丢弃 359 段，多为 ASR 转写异常的极短或超长文本；文本去重与音频质检仅丢弃 39 段，表明前序步骤已将绝大部分低质数据拦截，最终保留率达 86.7\%。

说话人聚类（表~\ref{tab:cluster-top15}）表现总体良好：KMeans（$n=30$）将 15,685 段的声纹特征自动聚合为 30 个簇，其中 24 个簇成功映射到 14 个剧中角色。一个值得注意的现象是，部分角色（如甄嬛）的语音片段分散在 5 个不同簇中——这是因为同一角色在不同剧情场景下的语气、情绪、语速差异较大，导致声纹特征在嵌入空间中形成了多个子簇，后期人工标注环节对这些子簇完成了跨簇合并。Top 15 簇中仅有 spk\_25（666 段、60.7 分钟）因 KMeans 错误合并了多个不同说话人的声音而最终淘汰，说明 KMeans 在多数情况下能够有效区分不同说话人，但对声纹特征接近的少数说话人存在混淆风险。

数据集按 95:5 划分训练/验证集后，导出为 CosyVoice Kaldi 格式（wav.scp / text / utt2spk / spk2utt / instruct），并进一步提取为 Parquet 格式供模型训练使用。该数据集格式通用，亦可适配 GPT-SoVITS、VITS、Fish-Speech 等主流语音合成框架。

\section{问题分析}
\label{sec:problems}

在项目实施过程中，团队遇到并解决了以下关键问题：

\subsection{CosyVoice3 微调过拟合}
\label{sec:prob-overfit}

如第~\ref{sec:cosyvoice}节所述，CosyVoice3-0.5B 全参数 SFT 微调在第 3 个 Epoch 出现严重过拟合，训练损失与验证损失出现显著背离（图~\ref{fig:loss-curve}）。这一问题的根本原因可归结为以下三个层面：

\textbf{(1) 梯度投票权不对等。} 14 角色联合训练时，甄嬛（29.5\%）、皇上（17.3\%）和皇后（10.6\%）三个角色合计占据约 57\% 的训练样本。模型每更新两步即有一歩被这三个角色主导，温实初（2.0\%，253 段）等小角色在整个训练过程中极少出现，其梯度信号被淹没在主导角色的梯度海洋中。

\textbf{(2) Speaker Embedding 条件信号退化。} CosyVoice3 的推理逻辑为 $\text{speaker\_embedding} + \text{text} \to \text{speech\_token}$，模型需学会根据输入声纹选择对应音色。然而，在 57\% 样本来自前三大角色的数据分布下，模型发现了一种捷径——\textit{部分忽略 Speaker Embedding，直接输出高频角色的音色特征}，因为这一策略在多数训练样本上 Loss 最低。后果是小角色的 Speaker Embedding 退化为无效信号，验证集上合成音频的音色偏向甄嬛或皇上。

\textbf{(3) 预训练表示被快速覆盖。} CosyVoice3-0.5B 的预训练权重编码了通用 TTS 能力，微调时主导角色率先改写关键表示层；小角色的少量数据无力争夺这些位置，只能附着在已被主导角色占据的表示空间之上，合成音色偏离原声是必然结果。

\subsection{GPT-SoVITS 训练与调优}
\label{sec:prob-gptsovits}

\textbf{小样本角色不稳定。} GPT-SoVITS 的角色级微调虽然避免了多角色混合训练带来的音色互相污染，但对单角色数据规模依然敏感。甄嬛、皇上等高频角色拥有数小时级语音，模型能够学习到较稳定的音色和韵律；而温实初仅 253 段、约 20.0 分钟，小样本角色的语音覆盖场景有限，训练后更容易出现音色偏薄、语速不自然或尾音发虚的问题。这说明“单角色独立微调”只能隔离角色间干扰，不能消除数据稀缺本身造成的泛化不足。

\textbf{版本效果存在角色差异。} 实验中 \texttt{v4} 与 \texttt{v2ProPlus} 在不同角色上的听感并非完全一致。\texttt{v4} 使用 LoRA 形式进行 SoVITS 微调，整体更适合作为默认版本；但在个别角色上，\texttt{v2ProPlus} 的发音稳定性或韵律连续性更好。因此系统没有将版本选择写死，而是在服务端保留 \texttt{v4} 主版本和 \texttt{v2ProPlus} 回退版本：默认请求优先使用 \texttt{v4}，当推理失败或指定版本不可用时自动切换到 \texttt{v2ProPlus}。这使网页演示不依赖单一权重，降低了现场运行风险。

\textbf{LoRA 约束与训练轮次取舍。} GPT-SoVITS \texttt{v4} 训练中采用 \texttt{lora\_rank=32}，目的在于限制可训练参数规模，使模型在角色数据量有限时不过度改写预训练表示。rank 过低时，模型对角色音色的适配不足，合成结果更接近通用预训练音色；rank 过高时，小角色容易把训练集中少数句子的韵律和噪声一并记住，出现过拟合式的僵硬表达。最终训练脚本统一保留第 10 个 epoch 的 GPT 与 SoVITS 权重，并清理非最终 checkpoint，在效果和工程可管理性之间取得折中。

\textbf{解决方案与经验。} GPT-SoVITS 路线最终形成了“单角色独立微调 + 双版本并行 + LoRA 约束 + 最终权重清理”的工程策略。该策略提升了训练可复现性和部署稳定性，但从主观听感看，微调方案仍受制于角色数据量、参考音频质量和模型对情绪控制的间接性。因此在最终系统中，GPT-SoVITS 更适合作为可对照、可回退的备用 TTS 引擎；主力实时合成则采用 CosyVoice3 Zero-shot 与情绪 Speaker 方案。

\subsection{情绪控制与音色保真的矛盾}
\label{sec:prob-emotion}

Zero-shot 方案（第~\ref{sec:cosyvoice}节）能够完美复现角色的中性音色，但面临一个核心矛盾：如何在不损失音色保真度的前提下实现自然的情感表达？

\textbf{Instruct 控制的局限。} CosyVoice3 提供了 Instruct 控制接口，可通过文字指令驱动模型以特定情绪说话。然而实验发现，Instruct 方式在施加情绪控制的同时，会对声纹嵌入的表示空间产生干扰，导致输出音色偏离参考说话人的原始特征——\textit{情绪控制与音色保真构成一对矛盾，难以兼顾。}GPT-SoVITS 引擎面临类似困境：其本身不支持稳定的情绪参数直接输入，只能通过替换参考音频来间接影响输出语气（参见第~\ref{sec:prob-gptsovits}节）。

\textbf{情绪 Speaker 方案。} 为解决上述矛盾，提出了一种创新的\textbf{情绪 Speaker}方案：不依赖 Instruct 指令或参数化情绪控制，而是从 76 集原剧音频中手工筛选角色在真实情绪状态下的台词片段，将这些情绪音频本身作为独立的 Zero-shot Speaker 注册（采集统计见表~\ref{tab:emotion-data}）。推理时，根据前端传入的 \texttt{character\_id} 和 \texttt{emotion} 字段查表路由到对应的情绪 Speaker，本质上是使用该角色在\textit{真实情绪状态}下的声纹特征进行语音合成——走纯 Zero-shot 路径，因此\textbf{音色完全不丢失}。

\textbf{方案效果。} 该方案有效解决了情绪控制与音色保真之间的矛盾：在喜悦、愤怒、悲伤、平静四种情绪上，合成音频的语气自然贴近剧中角色的真实情绪表达，同时保持了与中性 Speaker 一致的音色还原度。在 CosyVoice3 引擎上，该方案已成为系统默认的情绪控制策略。在 GPT-SoVITS 引擎上，同样采用情绪参考音频替代单一中性参考音频的方式实现情绪路由，但在音色一致性上略逊于 CosyVoice3 的 Zero-shot 方案（详见第~\ref{sec:prob-gptsovits}节）。

\section{总结与展望}
\label{sec:conclusion}

\subsection{项目成果}

本项目成功构建了《甄嬛传》多角色语音对话系统，主要成果如下：

\begin{itemize}[leftmargin=*]
    \item 自主构建了包含 14 个角色、13,605 段语音片段、总时长约 16.7 小时的高质量中文语音数据集。
    \item 实现了 CosyVoice3 Zero-shot（30 个 Speaker）与 GPT-SoVITS 微调（28 组权重）的双引擎语音合成。
    \item 创新性地提出了\textbf{情绪 Speaker}方案，通过 107 段手工采集的真实情绪音频实现了 4 种情感语音合成，有效解决了情绪控制与音色保真的矛盾。
    \item 实现了单人智能对话与双人即兴对戏两种交互模式，支持流式语音播放，首音延迟约 1.5 秒。
    \item 构建了完整的前后端系统，包含古风宫廷风格 UI、FastAPI 中间层和云端 GPU 服务。
\end{itemize}

\subsection{未来展望}

\begin{itemize}[leftmargin=*]
    \item 扩充更多角色（如端妃、敬妃、流朱等）和更多情绪类别。
    \item 探索更高效的微调策略（如单角色独立微调 + LoRA），解决数据不均衡下的过拟合问题。
    \item 引入数字人技术（SadTalker / MuseTalk），在算力允许时实现角色视频对话。
\end{itemize}
