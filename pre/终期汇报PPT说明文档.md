## 第 3 页 · 项目概述（一）：我们做了什么

### 一句话描述

构建了一个面向《甄嬛传》的 **AI 角色语音对话系统**，用户以"现代来客"或"宫廷中人"身份与剧中角色实时对话——角色不仅能用**本人在剧中的真实声音**回应用户，还能根据对话内容自动带上**喜悦/愤怒/悲伤/平静**四种情绪。

### 系统核心能力

| 能力 | 说明 |
|---|---|
| 🎤 **角色语音克隆** | 从 76 集原剧音频中提取并标注 14 个角色的声音，使用 CosyVoice3 和 GPT-SoVITS 双引擎驱动 |
| 🧠 **角色性格建模** | LLM 结合角色人设 Prompt（身份、性格、经典台词风格），按角色口吻生成对话回复 |
| 😊 **情感语音合成** | 支持喜悦/愤怒/悲伤/平静四种情绪，通过手工采集的真实情绪音频控制语气 |
| 💬 **多模式对话** | 单人对话（现代来客 / 宫廷中人）、双人即兴对戏（两个 AI 角色自动对话） |
| ⚡ **流式 TTS 输出** | SSE + Web Audio API 实现边生成边播放，首音延迟约 1.5 秒 |

### 技术栈概览

```
语音合成引擎： CosyVoice3-0.5B（Zero-shot + 流式） + GPT-SoVITS（微调）
大语言模型：   用于角色对话生成（单人 + 双人即兴对话）
语音识别：     faster-whisper-small（用户语音输入转文字）
前端：         React + Vite，古风宫廷 UI
后端：         FastAPI 中间层（yiping-backend）+ 模块化服务
部署：         AutoDL 云 GPU（RTX 4080 SUPER）+ SSH 隧道穿透
```

## 第 4 页 · 数据集构建（一）：从原始音频到角色数据

### 数据来源

- 数据源：《甄嬛传》全集 **76 集** MP3 音频（128kbps, 44.1kHz, Stereo）
- 预处理：已去片头片尾（无人播剧版），原始总时长约 **80 小时**
- 计算资源：租用 AutoDL / Gpufree 云 GPU 实例（RTX 4080 SUPER 单卡）

### 数据处理 Pipeline（五步全流程）

```
76 集 MP3 (44.1kHz Stereo, ~80h)
    │
    ▼ UVR5 demucs (mdx_extra_q 模型)           ← 人声分离
纯净人声 (44.1kHz Stereo)
    │
    ▼ FunASR SeacoParaformer (VAD + ASR)        ← 语音切分 + 转写
    │   时间戳合并（间隔 > 800ms 切分）
    ▼
15,685 段 WAV (16kHz Mono, 2~15s/段)
    │
    ▼ eres2netv2 → Speaker Embedding (512维)    ← 声纹提取
    │   + KMeans 聚类 (n_clusters=30)
    ▼
30 个匿名说话人簇 (spk_00 ~ spk_29)
    │
    ▼ 人工角色标注（关键词辅助 + 三轮迭代）      ← 说话人识别
    │   + 数据清洗与质量过滤
    ▼
最终数据集：14 角色，13,605 段，约 1,005 分钟（16.7 小时）
```

## 第 7 页 · GPT-SoVITS 微调

### 数据准备

使用与 CosyVoice 相同的 14 角色标注数据，转换为 GPT-SoVITS 标准 list 格式：

```
音频绝对路径|说话人|语言|文本
```

按角色拆分，每个角色保留 `all / train / dev` 三类文件，覆盖全部 14 角色。

### 微调核心策略

| 策略 | 说明 |
|---|---|
| **单角色独立微调** | 每个角色单独训练，不混合训练，降低音色混淆 |
| **双版本并行** | v4（主版本，质量较新）+ v2ProPlus（备选版本，个别角色效果不佳时回退） |
| **LoRA 微调** | SoVITS 声学部分使用 LoRA（`lora_rank=32`），降低过拟合风险 |
| **批量自动化** | 脚本自动发现角色 list，自动按角色和版本生成实验名，支持多 GPU 分片预处理 |

### 训练流程

```
角色 list
  → 1A 文本特征提取（中文 RoBERTa）
  → 1B HuBERT / SSL 特征提取（chinese-hubert-base）+ 32k 音频处理
  → 1C Semantic Token 提取（预训练 SoVITS 模型）
  → SoVITS 声学部分训练（v4: LoRA / v2ProPlus: 全量）
  → GPT 语义部分训练（s1v3.ckpt 预训练权重）
  → 导出最终权重（保留 Epoch 10）
```

### 训练规模

- 14 个角色 × 2 个版本 = **28 组独立权重**
- 每个角色独立训练，每个版本约保留 Epoch 10 的 GPT + SoVITS 权重
- 训练完成后清理中间 checkpoint，仅保留最终权重以减少磁盘占用

## 第 9 页 · CosyVoice Zero-shot 方案与情绪 Speaker

> **最终采用方案**：CosyVoice Zero-shot 的合成效果在实际听感测试中**明显优于** GPT-SoVITS 微调和 CosyVoice SFT 微调，音色还原度最高、发音最清晰自然，因此作为系统的主力 TTS 引擎。

### 为什么从微调转向 Zero-shot？

| 对比维度 | SFT 微调 | Zero-shot |
|---|---|---|
| 数据需求 | 每角色需大量小时级数据 | 每角色仅需 1 条参考音频（5~15s） |
| 训练时间 | 数小时（LLM + Flow + HiFiGAN） | **0 秒**（无需训练） |
| 音色质量 | 过拟合后下降，小角色更差 | **高**，预训练模型泛化能力强 |
| 新增角色 | 需重新训练全部模型 | 即插即用（添加参考音频即可） |
| 稳定性 | 依赖调参和算力 | 开箱即用 |

### 14 角色 Zero-shot 注册

从清洗后的 13,605 段数据中，每个角色精选 **1 条高质量参考音频**（5~10s）+ 对应台词文本，通过 `add_zero_shot_spk` 提取声纹特征并注册至 `spk2info.pt` 持久化保存。

### 情绪控制：从 Instruct 到情绪 Speaker

**问题**：Zero-shot 可完美复现角色中性音色，但无法自然表达喜悦/愤怒/悲伤等情绪。若通过 LLM 的 Instruct 控制情绪，音色会有明显损失——**情绪控制与音色保真是一对矛盾**。

**解决方案**：从 76 集原剧中**手工筛选** 4 个主角 × 4 种情绪的台词片段，用角色**真实情绪状态下的音频**做声纹提取，注册为独立情绪 Speaker：

| 角色 | 喜悦 | 愤怒 | 悲伤 | 平静 | 合计 |
|---|---|---|---|---|---|
| 甄嬛 | 3 段 | 6 段 | 5 段 | 5 段 | **19 段** |
| 华妃 | 8 段 | 14 段 | 13 段 | 6 段 | **41 段** |
| 皇上 | 3 段 | 5 段 | 1 段 | 3 段 | **12 段** |
| 皇后 | 5 段 | 11 段 | 11 段 | 8 段 | **35 段** |
| **合计** | **19** | **36** | **30** | **22** | **107 段** |

### 推理时的情绪路由

```
前端请求：{ character_id: "huafei", emotion: "愤怒", text: "贱人就是矫情！" }
    │
    ▼ TTS 服务器
    ├─ EMOTION_SPEAKER_MAP[("huafei", "愤怒")] → "huafei_anger" ✓
    ├─ zero_shot("贱人就是矫情！", spk_id="huafei_anger")
    └─ 输出华妃真实愤怒状态的语气 ✅ 音色完全不丢
```

### 最终 Speaker 注册总量

```
14 个基础角色 Speaker（中性音色）
+ 16 个情绪 Speaker（4 角色 × 4 情绪）
= 30 个 Zero-shot Speaker
```

## 第 10 页 · ASR + LLM 模块实现

### ASR 语音识别

| 配置项 | 设计 |
|---|---|
| 模型 | faster-whisper-small（基于 CTranslate2 加速） |
| 语言 | 强制中文 `zh`，避免短句误判 |
| VAD | 开启 `vad_filter=True`，自动过滤静音片段 |
| 模型加载 | 服务首次调用时加载并缓存为全局模型，避免每次请求重新加载 |
| 前端策略 | 识别结果回填输入框，用户确认或修改后再发送，降低误识别影响 |

> 选择 faster-whisper-small 的理由：在中文准确率、推理速度与显存占用之间取得最优平衡，适合短句实时对话场景。

### LLM 单人对话 Prompt 设计

LLM 通过分层 Prompt 一次生成展示文本、情绪标签和两套 TTS 文本，避免多次请求导致延迟上升：

```json
{
  "text": "角色回复（前端展示，1~3 句）",
  "emotion": "喜悦 / 愤怒 / 悲伤 / 平静",
  "tts_texts": {
    "cosyvoice": "带副语言控制 token（如 [breath][laughter]）",
    "gpt_sovits": "干净中文文本（仅标点控制停顿）"
  }
}
```

### 现代来客 vs 宫廷中人：身份感知对话

| 维度 | 现代来客 | 宫廷中人 |
|---|---|---|
| 用户身份描述 | "现代来客" | "宫廷中人：华妃 / 皇上…" |
| 对话关系 | 松散，解释性更强 | 有明确身份和权力关系 |
| 角色称呼 | 更泛化 | 受用户扮演角色影响 |
| 情绪张力 | 相对平缓 | 更容易出现试探、压迫、对峙 |

### 双人即兴对话 Prompt 设计

每轮将**完整对话历史 + 场景 + 双方角色人设**注入 user_prompt，LLM 仅生成当前发言角色的下一句台词（20~50 字）和情绪。后端循环调用 LLM → 交替发言角色 → TTS 合成，直到达到设定轮数或用户停止。

### 双 TTS 文本策略

| 引擎 | 文本处理 |
|---|---|
| CosyVoice | 允许 LLM 加入 `[breath][laughter][sigh]` 等副语言 token，后端校验白名单 + 限 2 个 |
| GPT-SoVITS | 自动清洗方括号 token、HTML 标签、压缩空白、补齐中文句末标点 |

```
│  │  xiao-asr_llm (FastAPI :8001)                 │   │
│  │  ├─ ASR: faster-whisper-small + VAD           │   │
│  │  ├─ POST /chat         单人对话生成            │   │
│  │  └─ POST /generate/duet  双人即兴对话         │   │
│  └──────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────┐   │
│  │  gpt-sovits-service (FastAPI :8002)           │   │
│  │  ├─ 14 角色 × 2 版本独立权重                  │   │
│  │  └─ 情绪参考音频路由                          │   │
│  └──────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
                      │ SSH 隧道
                      ▼
┌──────────────────────────────────────────────────────┐
│  本地 (Windows)                                      │
│  ├─ yiping-backend (:8000) → 路由 LLM + TTS         │
│  └─ yiping-frontend (:5173) → React UI + 流式播放   │
└──────────────────────────────────────────────────────┘
```

### yiping-backend 中间层接口

| 接口 | 功能 |
|---|---|
| `POST /asr` | 用户录音上传，返回识别文本 |
| `POST /chat` | 角色对话，返回回复文本 + 情绪标签 + 双 TTS 文本 |
| `POST /synthesize` | 语音合成（非流式） |
| `POST /synthesize/stream` | SSE 流式语音合成（CosyVoice 专用） |
| `POST /duet/start` | 双人即兴对话，SSE 逐轮推流 |
| `POST /summary` | 对话总结，返回态度评价 + 古风总结语 |

## 第 13 页 · 核心功能（一）：单人智能对话

### LLM Prompt 分层设计

单人对话的 Prompt 分为 system_prompt（定义输出格式和约束）和 user_prompt（注入角色、场景和用户身份），LLM 一次返回：

```json
{
  "text": "角色回复（前端展示）",
  "emotion": "喜悦 / 愤怒 / 悲伤 / 平静",
  "tts_texts": {
    "cosyvoice": "带副语言控制 token 的文本",
    "gpt_sovits": "干净中文文本（仅标点控制停顿）"
  }
}
```

### 现代来客 vs 宫廷中人

| 维度 | 现代来客 | 宫廷中人 |
|---|---|---|
| 用户身份 | 外部访客 | 剧中指定角色（如华妃、皇上） |
| 对话关系 | 松散，解释性更强 | 有明确身份和权力关系 |
| 角色称呼 | 更泛化 | 受用户扮演角色影响 |
| 情绪张力 | 相对平缓 | 更容易出现试探、压迫、对峙 |

### 对话页功能

- 🎙️ **语音输入**：浏览器 MediaRecorder API 录音 → ASR 识别 → 回填输入框（用户可修改后再发送）
- 💬 **快捷台词**：预设 4 个经典台词胶囊按钮（"臣妾做不到啊" / "贱人就是矫情" / "愿得一心人" / "皇上圣安"）
- 🎛️ **TTS 引擎切换**：GPT-SoVITS / CosyVoice 二选一，可对比合成效果
- 🎭 **情绪手动覆盖**：支持用户强制指定角色情绪（自动 / 喜悦 / 愤怒 / 悲伤 / 平静）
- 📸 **对话总结卡片**：结束对话后生成"今日宫廷密录"，含角色态度评价 + 古风总结语，支持一键导出 PNG

## 第 14 页 · 核心功能（二）：双人即兴对话

### 功能描述

用户选择两个角色 + 设定场景 → 两个 AI 角色自动你一句我一句地对话 → 语音流式播放，用户全程旁观。这是一个 **LLM 自动编剧 + TTS 实时演出**的创新功能。

### 技术实现

```
用户设定：角色 A (华妃) vs 角色 B (甄嬛) | 场景：御花园偶遇 | 先手：华妃
    │
    ▼ 循环生成（最多 N 轮，或用户手动停止）
    │
    ├─ 当前发言角色 = 华妃
    │   ├─ LLM 生成回复（注入角色人设 + 场景 + 完整历史）
    │   ├─ SSE 推送文本 → 前端展示气泡（高亮发言方）
    │   └─ TTS 流式合成 → Web Audio API 排队播放
    │
    ├─ 当前发言角色 = 甄嬛  ← 交换角色
    │   ├─ LLM 生成回复
    │   ├─ ...
    │   └─ TTS 流式合成
    │
    └─ 重复直到停止...
```

### 关键技术点

- 每轮传递**完整对话历史**，LLM 理解上下文后生成下一句
- 两个角色各自保留独立人设 Prompt，确保说话风格不混淆
- 文本展示与音频播放**同步进行**，形成"即兴对戏"效果
- 支持随时「停止」中断

## 第 15 页 · 核心功能（三）：流式 TTS 与 ASR

### CosyVoice SSE 流式 TTS

```
TTS Cloud (:8003)  →  SSH 隧道  →  yiping-backend (:8000)  →  SSE 推送  →  前端
                                                                              │
                                        每块 {i, audio(base64), dur}          ▼
                                                                   Web Audio API
                                                                   按 i 序号排队
                                                                   scheduleStartTime
                                                                       │
                                                                       ▼
                                                              无缝顺序播放
                                                              首音延迟 ~1.5s
                                                               RTF 0.5~0.8
```

### 推理性能

| 指标 | 非流式 | 流式 (SSE) |
|---|---|---|
| 首音延迟 | 2~3s | **1.5s** |
| RTF (Real-Time Factor) | 0.5~0.65 | 0.5~0.8 |

### ASR 语音输入设计

| 配置项 | 设计 |
|---|---|
| 模型 | faster-whisper-small（CTranslate2 加速） |
| 语言 | 强制中文 `zh` |
| VAD | 开启 `vad_filter=True`，自动过滤静音 |
| 模型加载 | 首次调用加载并缓存为全局模型 |
| 前端策略 | 识别结果回填输入框，用户确认后发送（降低误识别影响） |
| 模型选择理由 | 在中文准确率、推理速度、显存占用之间更均衡，适合短句实时对话场景 |

## 第 16 页 · 工程亮点总结

### 数据集构建

1. **全流程自主构建**：从原始 MP3 到 14 角色标注音频，未使用任何现成数据集，最终规模达 **13,605 段、16.7 小时**
2. **关键词辅助标注系统**：为 14 个角色设计 70+ 特征关键词，结合人工试听，三轮迭代完成说话人识别
3. **多模型串联**：UVR5（人声分离）→ FunASR（VAD+ASR）→ eres2netv2（声纹提取）→ KMeans（聚类），全流程自动化

### 模型训练与推理

4. **双引擎并行**：CosyVoice3 Zero-shot（30 Speaker）+ GPT-SoVITS 微调（14×2=28 组权重），用户可实时切换对比
5. **情绪 Speaker 创新方案**：手工采集 107 段真实情绪音频注册为独立 Speaker，用真实情绪替代 Instruct 控制，解决情绪与音色保真矛盾
6. **微调踩坑经验**：CosyVoice3-0.5B 14 角色联合微调因数据不均衡过拟合，果断转向 Zero-shot 保住交付质量

### 前后端与系统

7. **流式 TTS 链路**：SSE + Web Audio API 按序排队调度，首音延迟降至 1.5s
8. **双人即兴对话**：LLM 逐轮驱动双角色自动对话，完整历史传递，形成 AI 编剧 + TTS 演出
9. **身份感知对话**：现代来客 / 宫廷中人双身份系统，LLM 据此调整角色口吻、称呼和情绪张力
10. **全云端部署**：数据处理、模型训练、TTS 推理均在云 GPU 完成，本地零算力依赖（租用 AutoDL RTX 4080 SUPER）

## 第 18 页 · 总结与展望

### 项目成果

- 构建了《甄嬛传》**首个**多角色语音对话系统，覆盖 12 个可对话角色
- 自主构建 **14 角色、13,605 段、16.7 小时**高质量中文语音数据集
- 实现双 TTS 引擎（CosyVoice3 Zero-shot + GPT-SoVITS 微调），支持 30 个 Speaker
- 创新性地采用"情绪 Speaker"方案，用 107 段手工采集情绪音频实现 4 种情感语音合成
- 实现双人即兴对话、流式 TTS、ASR 语音输入等完整交互功能

### 未来可改进方向

- 扩充更多角色（如端妃、敬妃、流朱等）和更多情绪类别
- 探索更高效的微调策略（如单角色独立微调 + LoRA），解决数据不均衡下的过拟合
- 引入数字人技术（已调研 SadTalker / MuseTalk），在算力允许时实现角色"开口说话"视频
- 增加声纹鉴权功能，实现"只有本人声音才能触发对话"的安全机制
