# PPT 汇报内容：ASR+LLM 与 GPT-SoVITS 语音生成模块

> 用途：本文档保留适合放入 PPT 展示和答辩阐述的内容，采用第三人称表述，重点突出 Prompt 设计、ASR 方案、GPT-SoVITS 微调流程、情绪控制与系统亮点。

---

## 1. 模块定位与整体链路

本部分工作面向“可交互角色对话”和“角色语音生成”两个核心目标，将 ASR、LLM 和 GPT-SoVITS 串联成完整链路。

```text
用户文字/语音输入
  -> ASR 将语音转成文本
  -> LLM 生成角色回复、情绪和 TTS 文本
  -> GPT-SoVITS 根据角色权重与参考音频生成角色语音
  -> 前端播放角色音频
```

系统实现的关键能力：

- 用户既可以文字输入，也可以语音输入。
- LLM 能根据角色、人设、用户身份和对话历史生成回复。
- 回复结果包含展示文本、情绪标签和面向不同 TTS 的文本。
- GPT-SoVITS 使用微调后的角色权重与情绪参考音频生成语音。
- 同一网站中可以对比 GPT-SoVITS 与 CosyVoice 的合成效果。

## 2. ASR 模块设计

ASR 采用 `faster-whisper-small` 模型，模型文件位于：

```text
models/faster-whisper-small
```

`faster-whisper` 是基于 CTranslate2 的 Whisper 高性能推理实现，相比原版 Whisper 更适合部署在服务端做低延迟语音识别。

### 模型选择理由

| 方案 | 特点 |
|---|---|
| Whisper tiny/base | 速度快，但中文识别准确率偏弱 |
| Whisper medium/large | 准确率更高，但延迟和显存压力更大 |
| faster-whisper-small | 在中文准确率、速度、显存占用之间更均衡 |

当前项目是网页实时对话场景，用户每次输入通常是短句，因此 `small` 模型更适合在准确率和响应速度之间取得平衡。

### ASR 推理配置

| 配置项 | 设计 |
|---|---|
| 模型路径 | `models/faster-whisper-small` |
| 语言 | 固定为中文 `zh` |
| VAD | 开启 `vad_filter=True` |
| 模型加载 | 服务首次调用时加载，并缓存为全局模型 |
| 输入格式 | 接收浏览器录音文件，通常为 `audio/webm` |
| 输出结果 | 返回识别文本、语言和置信度 |

### 巧思设计

- **模型缓存**：ASR 模型只在首次调用时加载，避免每次录音识别都重新加载模型。
- **VAD 过滤**：通过 `vad_filter=True` 自动过滤静音片段，减少无效音频对识别结果的影响。
- **中文强约束**：指定 `ASR_LANGUAGE=zh`，避免短句输入时被误判为其他语言。
- **前端不自动发送**：ASR 结果先回填到输入框，用户确认或修改后再发送，降低误识别带来的对话错误。
- **文本输入不受影响**：如果 ASR 未配置，语音识别接口会明确报错，但文字聊天仍可继续使用。

## 3. LLM 单人对话 Prompt 构建

单人对话的 Prompt 分为两部分：

```text
system_prompt：规定模型身份、输出格式、字段约束和 TTS 文本要求
user_prompt：注入当前角色、用户身份、历史对话、用户输入和情绪选择
```

### system_prompt 设计

system prompt 将 LLM 定义为《甄嬛传》角色对话生成器，并强制只输出 JSON：

```json
{
  "text": "角色回复",
  "emotion": "喜悦/愤怒/悲伤/平静",
  "tts_texts": {
    "cosyvoice": "给 CosyVoice 的文本",
    "gpt_sovits": "给 GPT-SoVITS 的文本"
  }
}
```

核心约束：

- `text` 是前端展示文本，1 到 3 句。
- 不能包含旁白、舞台说明、表情符号、英文 token 或 HTML。
- `emotion` 必须从喜悦、愤怒、悲伤、平静中选择。
- 如果用户手动指定情绪，回复内容和 `emotion` 必须匹配该情绪。
- `tts_texts.cosyvoice` 可以包含少量 CosyVoice 控制 token。
- `tts_texts.gpt_sovits` 必须是干净中文文本，主要靠中文标点控制停顿。

这种结构让 LLM 一次生成展示文本、情绪和两套 TTS 文本，避免多次请求导致延迟上升。

## 4. 现代来客与宫廷中人 Prompt 差异

单人对话中，用户身份会被写入 user prompt。不同对话入口对应不同身份设定。

### 现代来客模式

当用户以现代人身份进入对话时，Prompt 中的用户身份为：

```text
用户身份：现代来客
```

这种设计适合开放式提问，例如询问剧情、角色感受、宫廷生活等。角色回复会更偏向“对外解释”和“与陌生来客交流”，不强制绑定宫廷身份关系。

Prompt 结构示意：

```text
对话角色：甄嬛
角色风格：聪慧克制，语气温婉却有锋芒，常含机心与分寸。
用户身份：现代来客
最近对话：...
用户本轮输入：娘娘今日心情如何？
用户指定情绪：自动判断
请生成贴合角色身份的回复。
```

### 宫廷中人模式

当用户选择扮演某个剧中角色时，Prompt 中的用户身份会变为：

```text
用户身份：宫廷中人：华妃
```

或：

```text
用户身份：宫廷中人：皇上
```

这种模式会显式告诉 LLM：用户不是普通现代访客，而是宫廷关系网中的某个角色。LLM 因此需要调整称呼、语气、权力关系和情绪张力。

Prompt 结构示意：

```text
对话角色：甄嬛
角色风格：聪慧克制，语气温婉却有锋芒，常含机心与分寸。
用户身份：宫廷中人：皇上
最近对话：...
用户本轮输入：你近来可还安好？
用户指定情绪：平静
请生成贴合角色身份的回复。
```

### 两种模式的区别

| 维度 | 现代来客 | 宫廷中人 |
|---|---|---|
| 用户身份 | 外部访客 | 剧中角色 |
| 对话关系 | 松散、解释性更强 | 有明确身份和权力关系 |
| 角色称呼 | 更泛化 | 会受用户角色影响 |
| 情绪张力 | 相对平缓 | 更容易出现试探、压迫、对峙 |
| 适用场景 | 普通聊天、剧情问答 | 角色扮演、沉浸式对话 |

## 5. 双人即兴对话 Prompt 构建

双人即兴对话采用“当前发言角色 + 对方角色 + 场景 + 完整历史”的 Prompt 结构。

### system_prompt 设计

双人对话的 system prompt 将 LLM 定义为《甄嬛传》双人即兴对话生成器，并约束：

- 只输出 JSON。
- 字段必须是 `text` 和 `emotion`。
- `text` 是当前发言角色的一次自然回复。
- 文本长度控制在 20 到 50 个汉字。
- 只写台词，不写角色名、引号、旁白、动作描写或舞台说明。
- 回复必须衔接最近一句对话，贴合场景，不能重复历史台词。
- `emotion` 必须是四类情绪之一。

### user_prompt 设计

每一轮生成都会传入：

```text
当前发言角色
当前角色风格
对方角色
对方角色风格
场景/话题
完整对话历史
```

Prompt 结构示意：

```text
当前发言角色：甄嬛
当前角色风格：聪慧克制，语气温婉却有锋芒，常含机心与分寸。
对方角色：华妃
对方角色风格：骄矜凌厉，爱自称本宫，语气强势，喜怒分明。
场景/话题：两人在御花园偶遇
完整对话历史：
华妃：本宫倒要看看，你还有什么话说。
甄嬛：娘娘言重了，臣妾不过实话实说。
请生成当前发言角色的下一句台词，并判断情绪。
```

### 两人对话如何连续生成

双人对话不是一次性生成整段剧本，而是逐轮生成：

1. 根据用户选择确定角色 A、角色 B、场景和先手角色。
2. 当前角色调用 LLM 生成一句台词和情绪。
3. 生成结果追加到 `history`。
4. 下一轮交换当前发言角色和对方角色。
5. 新一轮 Prompt 会携带更新后的完整对话历史。
6. 重复以上过程，直到达到设定轮数。

这种设计的优势：

- 每一句台词都能基于上一句自然衔接。
- 两个角色各自保留独立人设和说话风格。
- 对话历史持续传递，减少前后矛盾。
- 前端可以边生成边展示，形成“即兴对戏”的效果。

## 6. LLM 输出后的文本校验与双 TTS 文本策略

LLM 一次生成三类文本：

| 字段 | 用途 |
|---|---|
| `text` | 前端气泡展示 |
| `tts_texts.cosyvoice` | 给 CosyVoice 的精细控制文本 |
| `tts_texts.gpt_sovits` | 给 GPT-SoVITS 的干净中文文本 |

### CosyVoice 文本策略

CosyVoice 支持副语言 token，因此允许 LLM 在语义合适的位置加入：

```text
[breath] [laughter] [sigh] [cough] [lipsmack] [noise] <strong>...</strong>
```

后处理会限制：

- token 必须来自白名单。
- 每次最多保留 2 个副语言 token。
- `[laugh]` 会被规范为 `[laughter]`。
- 如果 token 文本和展示文本语义不一致，会回退到安全版本。

### GPT-SoVITS 文本策略

GPT-SoVITS 不支持 CosyVoice 风格 token，因此文本会被清洗为纯中文：

- 删除方括号 token。
- 删除 HTML 标签。
- 压缩空白。
- 自动补齐中文句末标点。

这样可以避免 GPT-SoVITS 把控制 token 当成普通英文读出来。

## 7. GPT-SoVITS 微调数据准备

GPT-SoVITS 微调使用已经清洗和角色标注后的剧集语音片段。数据被转换成 GPT-SoVITS 标准 list 格式：

```text
音频绝对路径|说话人|语言|文本
```

按角色拆分后保存为：

```text
dataset/gpt_sovits_lists/by_role/{role}_all.list
dataset/gpt_sovits_lists/by_role/{role}_train.list
dataset/gpt_sovits_lists/by_role/{role}_dev.list
```

覆盖 14 个角色：

```text
anlingrong, caoguiren, cuijinxi, guojunwang,
huafei, huanbi, huanghou, huangshang,
meizhuang, supeisheng, taihou, wenshichu,
yelanyi, zhenhuan
```

数据准备亮点：

- 每条音频都保留绝对路径，方便训练脚本直接读取。
- 每条音频绑定真实台词，推理时可反查参考音频文本。
- 按角色独立整理 list，便于后续做单角色微调。
- 保留 `all/train/dev` 三类文件，便于训练、验证和推理复用。

## 8. GPT-SoVITS 微调流程

GPT-SoVITS 微调采用批量脚本自动化完成，核心流程如下：

```text
角色 list
  -> 1A 文本特征提取
  -> 1B SSL / HuBERT 特征与 32k 音频处理
  -> 1C Semantic Token 提取
  -> SoVITS 声学部分训练
  -> GPT 语义部分训练
  -> 导出最终权重
```

### 1A 文本特征提取

使用 GPT-SoVITS 内置文本处理流程，对角色台词提取文本特征：

- 使用中文 RoBERTa 预训练模型。
- 生成 `2-name2text.txt`。
- 为后续 GPT 语义模型训练提供文本输入。

### 1B HuBERT / SSL 特征提取

对音频提取 SSL 表征并生成 32k 音频中间结果：

- 使用 `chinese-hubert-base`。
- 生成 `4-cnhubert` 和 `5-wav32k`。
- `v2ProPlus` 版本额外提取 speaker verification 相关特征。

### 1C Semantic Token 提取

使用预训练 SoVITS 模型提取语义 token：

- 生成 `6-name2semantic.tsv`。
- 该文件用于 GPT 语义模型训练。

### SoVITS 训练

SoVITS 负责声音和声学建模：

- `v4` 使用 `s2_train_v3_lora.py`。
- `v4` 训练中启用 LoRA 形式，`lora_rank=32`。
- `v2ProPlus` 使用对应的 `s2_train.py` 和 v2ProPlus 预训练权重。

### GPT 训练

GPT 负责文本到语义 token 的建模：

- 使用 `s1_train.py`。
- 使用 `s1v3.ckpt` 作为预训练 GPT 权重。
- 每个角色、每个版本单独训练。
- 最终保留第 10 个 epoch 的 GPT 权重。

## 9. GPT-SoVITS 微调策略与亮点

### 单角色独立微调

微调时采用每个角色单独训练的方式，而不是将所有角色混在一个模型中训练。

优势：

- 降低角色之间的音色混淆。
- 每个角色有独立权重，便于单独替换、回退和比较。
- 网站侧可以根据 `character_id` 直接选择对应角色模型。

### 双版本训练

每个角色同时训练两个版本：

| 版本 | 作用 |
|---|---|
| `v4` | 默认使用，质量较新，作为主版本 |
| `v2ProPlus` | 作为备选版本，个别角色效果不佳时可回退 |

这种设计避免系统依赖单一版本，提高了演示和调试的稳定性。

### 只保留最终权重

训练脚本在全部角色训练完成后：

- 清理中间训练 checkpoint。
- 删除非最终 epoch 的导出权重。
- 只保留 epoch 10 的 GPT 和 SoVITS 权重。

优势：

- 减少磁盘占用。
- 避免权重版本混乱。
- 网站接入时权重路径更清晰。

### 批量训练自动化

批量脚本支持：

- 自动发现角色 list。
- 自动按角色和版本生成实验名。
- 支持跳过已有预处理结果。
- 支持多 GPU 分片预处理。
- 支持一键训练 14 角色的 `v4` 和 `v2ProPlus`。

这使得 GPT-SoVITS 微调从“手动逐角色操作”变成“批量可复现流程”。

## 10. GPT-SoVITS 推理与情绪控制设计

GPT-SoVITS 推理并不是简单传入文本，而是由以下因素共同决定最终声音：

```text
角色微调权重
参考音频
参考音频真实文本
待合成文本
标点停顿
采样参数
```

### 情绪控制方式

GPT-SoVITS 本身不稳定支持直接传入 `emotion=喜悦/愤怒` 参数，因此采用“情绪参考音频”控制方式：

```text
角色权重 + 情绪参考音频 + 参考音频文本 + 待合成文本
```

当前整理了 107 条情绪参考音频，覆盖：

| 角色 | 情绪 |
|---|---|
| 甄嬛 | 喜悦、愤怒、悲伤、平静 |
| 华妃 | 喜悦、愤怒、悲伤、平静 |
| 皇上 | 喜悦、愤怒、悲伤、平静 |
| 皇后 | 喜悦、愤怒、悲伤、平静 |

推理时根据角色和情绪随机选择一条参考音频，并反查对应台词作为 `ref_text`。这样生成音频会更接近该角色在真实情绪下的语气、语速和停顿。

### 参考音频质量控制

GPT-SoVITS 对参考音频时长比较敏感，因此加入了质量过滤：

- 优先选择 3 到 10 秒之间的参考音频。
- 自动跳过过短或过长的样本。
- 情绪样本不可用时，回退到角色 list 中的合法音频。

### 推理服务化亮点

GPT-SoVITS 被封装为常驻推理服务后，具备：

- 按角色和版本缓存模型，避免重复加载权重。
- 支持 `v4` 和 `v2ProPlus` 回退。
- 输出音频和 metadata，便于复现实验结果。
- 对输入文本自动清洗，避免读取无效 token。

## 适合 PPT 强调的最终亮点

1. **Prompt 分层设计**：system prompt 约束输出格式，user prompt 注入角色、人设、身份、历史和情绪。
2. **身份感知对话**：现代来客与宫廷中人使用不同身份描述，使回复关系和语气更准确。
3. **双人即兴对话**：每轮传递完整历史并交换发言角色，实现连续自动对戏。
4. **ASR 低延迟设计**：使用 `faster-whisper-small`，结合模型缓存、中文约束和 VAD 过滤。
5. **GPT-SoVITS 批量微调**：14 个角色、2 个版本、自动化预处理与训练。
6. **角色独立权重**：减少角色音色混淆，便于按角色独立部署和回退。
7. **情绪参考音频控制**：利用真实剧集情绪片段增强语气表现。
8. **双 TTS 文本策略**：CosyVoice 使用 token，GPT-SoVITS 使用干净文本和标点控制。
