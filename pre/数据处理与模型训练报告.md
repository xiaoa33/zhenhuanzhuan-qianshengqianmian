# 甄嬛传角色语音克隆 —— 数据处理与模型训练报告

> 本报告覆盖我从零构建 14 角色语音数据集、微调 CosyVoice3-0.5B 模型、以及最终采用 Zero-shot + 情绪控制方案的全过程。

## 二、数据处理 Pipeline（5 步）

### Step 0 — 环境准备

**服务器**：AutoDL / Gpufree 云 GPU 实例（RTX 4080 SUPER 单卡）
**依赖**：FunASR, PyTorch, ONNXRuntime, soundfile, librosa, scikit-learn, tqdm

### Step 1 — UVR5 人声分离 + VAD 切分 + ASR 转写 + 说话人聚类

```
76 集 MP3 (44.1kHz stereo)
    │
    ▼ UVR5 demucs (mdx_extra_q 模型)
纯净人声 (44.1kHz stereo)
    │
    ▼ FunASR SeacoParaformer (VAD + ASR 二合一)
    │   + 时间戳合并（间隔 >800ms 切分）
    ▼
15,685 段 WAV (16kHz mono, 2~15s/段)
    │
    ▼ eres2netv2 提取 Speaker Embedding (512维, 15,685段)
    │   + KMeans 聚类 (n_clusters=30)
    ▼
30 个匿名说话人簇 (spk_00 ~ spk_29)
```

**聚类结果 Top 15**：
| 簇 | 片段数 | 时长 |
|---|---|---|
| spk_06 | 1,241 | 77.0 min |
| spk_22 | 1,220 | 90.8 min |
| spk_10 | 877 | 61.6 min |
| ... | ... | ... |

### Step 2 — 人工角色标注

从每个说话人簇抽取 8 个样本，人工试听 + 结合**角色关键词字典**（如"朕/赐死/养心殿"→ 皇上，"贱人/矫情/一丈红"→ 华妃），将 30 个簇映射到剧中角色。

**三轮迭代**：第一轮 22/30 映射，第二轮 27/30，第三轮 24/30。最终确认 17 个角色后，**淘汰 3 个不可靠簇**（阿晋、净白师太、齐妃因 KMeans 不准确混入多人声音），保留 **14 个角色**。

**关键词辅助系统示例**：
```python
CHARACTER_KEYWORDS = {
    "甄嬛": ["臣妾", "本宫", "莞莞", "碎玉轩", "永寿宫"],
    "华妃·年世兰": ["贱人", "矫情", "年氏", "翊坤宫", "赏你一丈红"],
    "皇上·爱新觉罗·胤禎": ["朕", "赐死", "爱卿", "御前", "养心殿"],
    ...
}
```

### Step 3 — 数据清洗 & 质量过滤

| 过滤步骤 | 效果 |
|---|---|
| 角色过滤（去除非目标角色） | 15,685 → 14,003 (丢弃 1,682) |
| 文本过滤（空/过短/过长） | 14,003 → 13,644 (丢弃 359) |

**最终数据集**：**13,605 段，14 角色，约 1,005 分钟（16.7 小时）**

### Step 4 — 构建 CosyVoice Kaldi 格式

输出 `wav.scp`, `text`, `utt2spk`, `spk2utt` + `instruct`（角色语气指令），训练集 95%（12,925 段）+ 验证集 5%（680 段）。

### Step 5 — 提取 Embedding + Speech Token → Parquet

使用 CosyVoice3 内置的 campplus ONNX 和 speech_tokenizer_v3 ONNX 提取持续表征，输出为 Parquet 格式供训练加载。

## 三、CosyVoice3-0.5B 微调

### 环境搭建（大量兼容性问题）

在 AutoDL RTX 4080 SUPER 服务器上，遇到了以下关键问题并逐一解决：

| 问题 | 解决方案 |
|---|---|
| 新版 setuptools 移除 `pkg_resources` | 降级 `pip install setuptools==69.5.1` |
| `pip install matcha-tts` 把 Torch 从 2.3.1 升到 2.12 | 必须用内置 `third_party/Matcha-TTS` + `--no-deps` |
| 多个文件 `import whisper` 崩溃（无 Whisper 环境） | 改为 try/except 包裹（4 个文件） |
| `executor.py` 空 DataLoader 时 KeyError | 在 for 循环前加 `info_dict["tag"] = "TRAIN"` |
| Qwen tokenizer 路径缺失 | 指定 `CosyVoice-BlankEN` 目录 |
| `data.list` 路径错误拼接 | 改为相对路径（`../data/...`） |

### 训练配置

| 参数 | 值 |
|---|---|
| 预训练模型 | Fun-CosyVoice3-0.5B |
| GPU | RTX 4080 SUPER ×1, 16GB VRAM |
| 学习率 | 1e-5（预训练的 1/100） |
| 调度器 | ConstantLR |
| 最大 Epoch | 20（实际跑 5 轮） |
| Batch Size | 800 frames (梯度累积 ×2) |
| AMP | 启用 |
| 训练数据 | 12,925 条 (14 角色) |

### 训练过程（LLM 模型）

| Epoch | Train Loss | Train Acc | CV Loss | CV Acc |
|---|---|---|---|---|
| 0 | 1.64 → 1.69 | ~20% | 3.59 | 20.1% |
| 1 | 1.59 → 1.72 | ~23% | 3.59 | 20.3% |
| 2 | 1.58 → 1.38 | ~28% | 3.68 | 19.6% |
| 3 | 1.23 → 1.08 | ~33% | **4.05** | 18.2% |
| 4 | 0.73 → 0.86 | ~49% | **4.59** | 16.8% |
| 5* | 0.61 → 0.28 | ~62% | — | — |

> ⚠️ 从 Epoch 3 开始，Train Loss 持续下降但 CV Loss 明显上升，出现了**严重过拟合**。

### 微调结论

**SFT 微调结果不理想**：
1. CV Loss 在 Epoch 3 后反转上升（3.59 → 4.59），模型过度记忆训练集
2. 14 角色联合微调导致角色间音色混淆（小角色如温实初仅 20 分钟数据，不足支撑独立建模）
3. 合成测试时部分角色发音含混、音色偏离原声
4. 训练成本高，但最终效果不及预期

## 四、转向 Zero-shot 方案

### 为什么转 Zero-shot？

| 对比维度 | SFT 微调 | Zero-shot |
|---|---|---|
| 数据需求 | 每角色 ≥ 大量小时 | 每角色仅需 1 条参考音频 (5~15s) |
| 训练时间 | 数小时（LLM+Flow+HiFiGAN） | 0 秒（无需训练） |
| 音色质量 | 过拟合后下降，小角色更差 | **高**，预训练模型泛化能力强 |
| 新增角色 | 需重新训练 | 即插即用（添加参考音频即可） |
| 稳定性 | 依赖调参和算力 | 开箱即用 |

### Zero-shot 原理

CosyVoice3 预训练了 Speaker Embedding 提取网络（campplus），给定任意一段角色音频，自动提取声纹特征，在推理时注入 LLM 和 Flow Matching 模块，实现零样本声音克隆。

### 14 角色 Zero-shot 注册

从清洗后的 13,605 段数据中，每个角色精选 **1 条高质量参考音频** + 对应的台词文本，通过 `add_zero_shot_spk` 注册到 `spk2info.pt`，持久化保存。

**注册格式**：
```python
cosyvoice.add_zero_shot_spk(
    f"You are a helpful assistant.<|endofprompt|>{prompt_text}",
    wav_path,   # 5~10s 的角色原声片段
    spk_id      # 如 "华妃"
)
cosyvoice.save_spkinfo()  # 持久化到 spk2info.pt
```

## 五、人工采集情绪数据集

### 为什么需要情绪数据？

Zero-shot 可以完美复现角色的**中性音色**，但无法自然地表达**喜悦/愤怒/悲伤**等情绪。加入 LLM 的 Instruct 控制后，虽然可以指挥模型用特定情绪说话，但音色会有明显损失——**情绪控制和音色保真是一对矛盾**。

### 解决方案：情绪 Speaker

从原始剧集音频中**手工筛选** 4 个主角（甄嬛、华妃、皇上、皇后）× 4 种情绪（喜悦、愤怒、悲伤、平静）的台词片段，每个情绪注册为独立的 Zero-shot Speaker——本质是用角色**真实情绪状态的音频**做声纹提取，推理时不同情绪用不同 Speaker，走的是纯 Zero-shot 路径，**音色完全不丢**。

### 情绪数据采集

从 76 集原声中手动截取了大量情绪片段：

| 角色 | 喜悦 | 愤怒 | 悲伤 | 平静 | 合计 |
|---|---|---|---|---|---|
| 甄嬛 | 3 段 | 6 段 | 5 段 | 5 段 | **19 段** |
| 华妃 | 8 段 | 14 段 | 13 段 | 6 段 | **41 段** |
| 皇上 | 3 段 | 5 段 | 1 段 | 3 段 | **12 段** |
| 皇后 | 5 段 | 11 段 | 11 段 | 8 段 | **35 段** |
| **合计** | **19** | **36** | **30** | **22** | **107 段** |

每段用 FunASR 转写为实际台词文本（保证 prompt_text 和 audio 内容一致），再注册为 `{character}_{emotion}` 格式的 Speaker：
```
zhenhuan_joy, zhenhuan_anger, zhenhuan_sad, zhenhuan_calm
huafei_joy, huafei_anger, huafei_sad, huafei_calm
huangshang_joy, huangshang_anger, huangshang_sad, huangshang_calm
yixiu_joy, yixiu_anger, yixiu_sad, yixiu_calm
```

### 推理时的智能路由

```
前端请求 { character_id: "huafei", emotion: "愤怒", text: "贱人就是矫情！" }
  │
  ▼ TTS 服务器
  ├─ EMOTION_SPEAKER_MAP[("huafei", "愤怒")] = "huafei_anger" ← 命中！
  ├─ zero_shot("贱人就是矫情！", spk_id="huafei_anger")      ← 纯音色
  └─ 输出华妃真实愤怒状态的语气 ✅ 音色不丢
```

## 七、关键创新点

1. **全流程自主构建数据集**：从原始 MP3 到 14 角色标注音频，未使用任何现成数据集，数据规模达 16.7 小时、13,605 段
2. **关键词辅助标注系统**：利用剧本台词特点，为 14 个角色总共设计了 70+ 特征关键词，加速人工标注
3. **探索了微调 + Zero-shot 两条路线**：微调因数据不平衡过拟合后，果断转向 Zero-shot，保住了交付质量
4. **情绪 Speaker 方案**：从原剧手工采集 107 段情绪音频，用真实情绪注册替代 Instruct 控制，解决情绪表达与音色保真的矛盾
5. **流式 TTS**：SSE + Web Audio API 实现边生成边播放，首音延迟大幅降低
6. **全云端部署**：数据处理 + 训练推理均在云 GPU 完成，本地零算力依赖
