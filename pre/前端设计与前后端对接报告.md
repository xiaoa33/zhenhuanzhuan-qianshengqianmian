## 二、前端设计

### 2.1 技术栈

| 项目 | 选型 |
|------|------|
| 框架 | React（Vite） |
| 样式 | 纯 CSS（CSS Variables + Keyframes 动画） |
| 路由 | React Router |
| 截图导出 | html2canvas |
| 字体 | Ma Shan Zheng（毛笔风，Google Fonts）+ 宋体 |

### 2.2 视觉风格

确立了一套**宫廷古风**色彩体系，贯穿全部页面：

| 设计项 | 规格 |
|--------|------|
| 主色调 | 深墨绿 `#1a2a1a` / 深红褐 `#2a1a1a` |
| 强调色 | 宫廷金 `#c9a84c` |
| 文字色 | 米白 `#f5e6c8` |
| 卡片质感 | 半透明磨砂玻璃（`backdrop-filter: blur(8px)`）+ 金色细边框 |
| 标题字体 | Ma Shan Zheng 毛笔体 |
| 正文字体 | 宋体 / SimSun |

### 2.3 素材采集

所有素材均为**自行搜集**，放置于 `yiping-frontend/resource/`：

- **12 张角色剧照**：甄嬛、华妃、宜修、眉庄、安陵容、苏培盛、叶澜依、崔槿汐、温实初、浣碧、皇上、果郡王
- **11 张场景背景图**：主页背景、角色选择页背景，以及每个对话角色对应的专属场景图（部分角色共用同一场景背景）
- **背景音乐**：《长相思》（甄嬛传背景音乐），全局循环播放，音量 0.4，切换页面不中断

### 2.4 主页（HomePage）

- 全屏背景图叠加半透明遮罩
- **Ken Burns 效果**：背景图缓慢放大（scale 1→1.08，15 秒循环），营造电影感

```css
@keyframes kenBurns {
  from { transform: scale(1); }
  to   { transform: scale(1.08); }
}
```

- 中央：系统标题「甄嬛传·千声千面」（毛笔体）+ 副标题「入宫廷，闻千声」
- 右下角浮动音乐控制按钮（♪），点击切换静音/播放

### 2.5 角色选择页（CharacterSelectPage）

**用户身份二选一**：

| 身份 | 说明 |
|------|------|
| 现代来客 | 以现代人视角入宫与角色对话 |
| 宫廷中人 | 扮演剧中某个角色参与对话，LLM 据此调整人物口吻和称呼 |

选择「宫廷中人」后，展开第二层卡片列表，让用户选择自己要扮演的角色。

**12 个对话角色卡片**，每张含：角色剧照（220px，圆角裁切）、角色名（毛笔体）、一句经典台词（斜体）。

**悬停 / 点击效果**：悬停时卡片放大 1.05 倍 + 边框变亮金色；点击后 scale(1.08) 再淡出跳转。

### 2.6 对话页（ChatPage）

**布局**：左侧 1/3 角色展示区 + 右侧 2/3 对话区；移动端纵向折叠。

**角色展示区**：
- 显示当前角色剧照，金色边框，圆角 16px
- 语音播放期间显示声波动画（三根跳动竖线，纯 CSS 实现）
- 如后端返回数字人视频，`<video>` 标签替换剧照，播放完自动恢复静态图

**背景图动态切换**：根据对话角色映射到对应场景图，切换时 opacity 0→1 过渡 0.5s。

**情绪视觉反馈**（LLM 返回情绪标签后触发）：

| 情绪 | 视觉效果 |
|------|---------|
| 愤怒 | 对话区背景闪烁红色遮罩，持续 0.5s |
| 悲伤 | 背景整体亮度降低 10%（`filter: brightness(0.9)`） |
| 喜悦 | 背景短暂出现金色光晕 |
| 平静 | 无特殊效果 |

**对话气泡**：
- 角色发言左对齐（金色细边框 + 半透明深色背景），用户发言右对齐
- 古风圆角矩形（角色：`4px 16px 16px 16px`，用户：`16px 4px 16px 16px`）
- 角色气泡左侧附 32px 圆形头像

**底部输入区**：
- 文本输入框（金色边框，placeholder「请说话…」）
- 发送按钮「传话」（毛笔体，金色背景）
- 经典台词快捷按钮（4 个胶囊）：`臣妾做不到啊` / `贱人就是矫情` / `愿得一心人` / `皇上圣安`
- 麦克风按钮：调用浏览器 MediaRecorder API 录音，发送至 `/asr` 识别后填入输入框
- TTS 引擎选择器（GPT-SoVITS / CosyVoice）
- 情绪手动覆盖选择器（自动 / 喜悦 / 愤怒 / 悲伤 / 平静）

**流式音频播放**（CosyVoice 引擎）：
- 通过 SSE（Server-Sent Events）从后端逐块接收 base64 音频
- 利用 Web Audio API 的 `AudioContext` 按时间戳顺序排队播放，实现边生成边播放

### 2.7 对话总结卡片（SummaryCard）

点击「结束对话」弹出模态弹窗，内容：

1. 标题「今日宫廷密录」
2. 角色头像 + 角色名 + 共交谈轮数
3. 角色对用户的态度（LLM 返回，4 字以内，如「戒备有加」）
4. 风格化总结语（LLM 返回，20 字以内，古风口吻）
5. 「保存图片」（html2canvas 截图导出 PNG）/ 「再聊一次」（清空重开）

### 2.8 即兴对话页（DuetPage）

新增功能：让**两个剧中角色**基于给定场景自动对话，用户旁观。

- 选择两个角色 + 输入场景背景 + 指定开场白
- 后端驱动双角色轮流调用 LLM 生成回复，SSE 推送
- 前端顺序播放每句台词的合成音频（Web Audio API 排队）
- 有「停止」按钮可随时中断

## 三、数字人方案探索

### 3.1 调研与尝试

原计划将角色剧照 + 合成音频驱动成说话视频（数字人效果），调研了两个开源方案：

| 方案 | 特点 | 问题 |
|------|------|------|
| SadTalker | 纯本地运行，单 GPU 可跑 | Windows 环境配置繁琐，生成速度极慢（约 30s/句） |
| MuseTalk | 效果更自然 | 依赖更重，本地运行同样很慢 |

同时调研了商业 API（如 HeyGen、D-ID），但费用过高，不适合课程项目。

### 3.2 实际接入情况

成功在 Windows 下配置了 SadTalker 独立 Python 3.8 环境（主项目用 Python 3.13），解决了 lmdb 版本兼容问题，在 `yiping-backend/digital_human/sadtalker.py` 中实现了 subprocess 调用封装，后端 `/digital-human` 接口可正常调用。

但实际体验测试后，每轮对话需额外等待约五六分钟生成视频，严重影响对话流畅度，**最终决定放弃数字人功能**，改为：

- 对话期间展示静态角色剧照 + 声波动画
- 语音播放体验本身已经足够流畅（CosyVoice 流式首音延迟约 1.5s）

## 四、后端设计与对接

### 4.1 整体架构

```
前端 React (:5173)
    │
    └── yiping-backend (:8000)   ← 我负责
            │
    ┌───────┼──────────────┐
    ▼       ▼              ▼
模块A      模块B         SadTalker
ASR+LLM   TTS语音合成    数字人（最终未用）
:8001      :8002         subprocess
```

`yiping-backend` 作为中间层，负责：HTTP 接口暴露、请求路由、音频/视频静态文件托管、Mock 数据管理、SadTalker subprocess 封装。

### 4.2 接口设计

| 接口 | 功能 |
|------|------|
| `POST /asr` | 用户录音上传，返回识别文本 |
| `POST /chat` | 角色对话，返回回复文本 + 情绪标签 + TTS 文本 |
| `POST /synthesize` | 语音合成，返回音频 URL |
| `POST /synthesize/stream` | SSE 流式语音合成（CosyVoice 专用） |
| `POST /digital-human` | 数字人视频生成（已接入但未启用） |
| `POST /summary` | 对话总结，返回态度评价 + 总结语 |
| `POST /duet` | 即兴双角色对话，SSE 推流 |

### 4.3 Mock 策略

外部模块（LLM、TTS）开发过程中不同步完成，后端从第一天起就可以独立运行：

- 通过 `.env` 中 `USE_MOCK=true/false` 一键切换真实调用与 Mock
- Mock 数据（每角色 5 条预设回复、固定总结模板）集中在 `mock/mock_data.py`
- 切换无需修改任何业务代码，只需改环境变量并重启

### 4.4 外部模块接口规范

为 LLM 组（模块A）和 TTS 组（模块B）编写了完整接口规范文档（`docs/外部模块接口规范.md`），包含：

- 请求/响应 JSON 字段逐一说明
- emotion 枚举值（喜悦 / 愤怒 / 悲伤 / 平静）与双方对齐
- FastAPI 骨架代码，可直接作为开发起点
- 对接步骤（修改 `.env` → 重启后端）

TTS 接口支持两种响应格式（绝对路径 / base64），后端两种都能自动识别，降低对接摩擦。

### 4.5 流式音频实现（与 TTS 组联合）

CosyVoice 引擎支持流式 TTS，实现了完整的 SSE 流式链路：

```
TTS 服务器（AutoDL 云端 :8003）
    │ SSH 隧道 (:8002→:8003)
    ▼
yiping-backend (:8000) /synthesize/stream
    │ SSE，每块 JSON: {i, audio(base64), dur}
    ▼
前端 Web Audio API
    │ 按 i 排队，每块解码后 scheduleStartTime
    ▼
无缝顺序播放（首音延迟约 1.5s，RTF 0.5~0.8）
```

## 五、关键技术点

1. **现代/古代双身份系统**：用户身份通过 React Router state 在页面间传递，`/chat` 接口透传 `user_identity`/`user_role_id`/`user_role_name`，LLM 据此调整角色口吻与称呼
2. **情绪联动**：LLM → 情绪标签 → 前端视觉反馈（背景动画） + TTS 音色路由（不同情绪选不同 Zero-shot Speaker）
3. **流式播放**：SSE + Web Audio API `AudioContext` 排队调度，解决网络分块乱序问题
4. **浏览器录音**：`MediaRecorder` API 录音，`Blob` 以 `multipart/form-data` 上传 `/asr`
5. **截图导出**：`html2canvas` 将总结卡片 DOM 渲染为 PNG，用户可一键保存对话记录

## 六、遇到的主要问题与解决

| 问题 | 解决方案 |
|------|---------|
| 数字人生成延迟高（~30s/轮），体验差 | 放弃数字人，改为静态剧照 + 声波动画 |
| SSE 音频块乱序导致播放杂音 | Web Audio API 按块序号 `i` 计算 `startTime` 排队，不依赖到达顺序 |
| LLM/TTS 模块开发不同步 | Mock 策略，后端从第一天可独立运行，不互相阻塞 |
| 浏览器录音格式（webm）ASR 模块不支持 | 接口规范明确前端发 `audio/webm`，由 ASR 模块内部转格式处理 |
| Windows Python 多版本冲突（SadTalker 需 3.8，主项目 3.13） | SadTalker 独立 venv，`subprocess` 调用时指定 `SADTALKER_PYTHON` 路径 |
